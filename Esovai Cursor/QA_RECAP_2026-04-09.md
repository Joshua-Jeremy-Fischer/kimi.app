## 1. Behobene Bugs dieser Session

### Bug 1 — Job-Crawler: LLM antwortet mit `tool_calls` statt Text
**Symptom:** Alle drei Job-Profile zeigten `"LLM lieferte eine leere Antwort (200 OK, aber kein content)."` im Postfach.

**Ursache:** Kimi K2.5 interpretierte den Job-Crawler-Prompt als Agenten-Kontext und antwortete mit `finish_reason: tool_calls` statt Text — obwohl keine Tools übergeben wurden.

**Fix (3 Stufen, `backend/job-crawler.js`):**

| Stufe | Was | Wo |
|---|---|---|
| Fix 1 | `createCompletion()` Helper: versucht `tools: [], tool_choice: "none"`, Fallback ohne bei 400-Fehler | Z. 216–229 |
| Fix 2 | Bei `tool_calls` im Response: Follow-up User-Message "Du hast KEINE Tools" | Z. 258–274 |
| Fix 3 | `/no_think` Präfix im System-Prompt + `stripThink()` für `<think>`-Tags | Z. 179–191 |

**Status:** Deployed. `tool_calls=false` in Logs bestätigt — Hauptproblem behoben.

---

### Bug 2 — Monitor: `⚠️ Disk Usage: NaN%` / `Noch undefined frei`
**Symptom:** Postfach-Einträge zeigten `NaN%` und `undefined` statt echten Werten.

**Ursache:** `df --output=pcent,avail` ist GNU-spezifisch. Der Alpine/BusyBox-Container kennt `--output` nicht — der Befehl scheiterte lautlos, `parseInt("")` → `NaN`.

**Fix (`backend/monitor.js:55–67`):**
```js
// Vorher (GNU-only):
"df -h /data --output=pcent,avail 2>/dev/null | tail -1"

// Nachher (BusyBox-kompatibel):
"df -h /data 2>/dev/null | tail -1"
// parts[4] = Use%, parts[3] = Avail
```
Zusätzlich: `Number.isFinite()`-Guard — bei Parse-Fehler wird kein Alert mehr gepostet (statt `NaN%`-Alert).

**Status:** Deployed. Tritt nach nächstem 10-min Monitor-Intervall nicht mehr auf.

---

### Bug 3 — Postfach-Titel: `"1 Stelle gefunden"` obwohl Content nur Fehlermeldung
**Symptom:** Titel zeigte `"💼 IT Security: 1 Stelle gefunden"` — Inhalt war aber `"LLM lieferte eine leere Antwort..."`.

**Ursache:** Der Zeilen-Zähler (`lines.filter(l => !l.includes("Keine"))`) filterte nur `"Keine..."` heraus, nicht LLM-Fehlermeldungen.

**Fix (`backend/job-crawler.js:310–336`):**
```js
const isErrorLike =
  /^LLM\b/i.test(raw) || /leere Antwort/i.test(raw) || /^Fehler\b/i.test(raw);
const isNoResultsLike =
  /^Keine Suchergebnisse gefunden\./i.test(raw) ||
  /^Keine passenden Stellen gefunden\./i.test(raw);

const lines = (isErrorLike || isNoResultsLike) ? [] : raw.split("\n").filter(...);
```
Regex deckt auch BOM/Zero-Width-Chars am String-Anfang ab.

**Status:** Deployed und verifiziert.

---

### Bug 4 — Job-Crawler: `finish=length` → leerer Content (ausstehend)
**Symptom:** Logs zeigen `finish=length tool_calls=false` — Modell wird durch `max_tokens` abgeschnitten bevor eigentlicher Text beginnt.

**Ursache:** Kimi schreibt `<think>...</think>` Blöcke vor der Antwort. Bei `max_tokens: 1800` wird das Modell mitten im Thinking-Block abgeschnitten. Nach `stripThink()` ist `content` leer.

**Fix (bereits in aktuellem Code, `backend/job-crawler.js:179–191`):**
- `/no_think` Präfix im System-Prompt unterdrückt CoT
- `stripThink()` als Fallback

**Status:** Fix deployed. Nächster vollständiger Crawler-Lauf wird zeigen ob `finish=length` weiter auftritt. Falls ja: `max_tokens` auf `3200/2400/1600` erhöhen als nächster Schritt.

## 4. Offene Punkte

### Kritisch (blockiert echte Job-Ergebnisse)
- **Kein Search-API-Key**: Ohne Tavily/Serper/Brave-Key läuft der Crawler nur mit SearXNG. Ergebnisqualität schwankend. Fix: einen Key in `.env` auf dem Server eintragen.
  ```env
  TAVILY_API_KEY=tvly-...
  # oder
  SERPER_API_KEY=...
  ```

### Mittel
- **`finish=length` bei IT Security**: Kimi schreibt zu viel Thinking-Kontext für die aktuellen `max_tokens`-Limits. Falls `/no_think` nicht greift: `max_tokens` auf `3200/2400/1600` erhöhen.
- **Stündliche Check-in-Nachrichten**: `push.js` schickt sich jede Stunde eine Inbox-Nachricht. Sollte per `ENABLE_CHECKIN_MESSAGES=false` ENV-Variable deaktivierbar sein.
- **eso-bot Tests nicht verifiziert**: `integration-test.mjs` existiert, wurde nie auf dem Live-Server ausgeführt. Test A (Datei schreiben/lesen) und Test B (web_fetch) ausstehend.

### Klein
- **Alte Postfach-Einträge**: Einträge mit `NaN%` und `"1 Stelle gefunden"` (Fehlermeldung) sind noch im Postfach sichtbar — manuell über das UI löschen.
- **`jobStore.running` Race-Condition**: Manueller Trigger während laufendem Crawl → `"Läuft bereits, überspringe"` ohne Feedback über Abschluss. Kein echter Bug, aber UX-suboptimal.
